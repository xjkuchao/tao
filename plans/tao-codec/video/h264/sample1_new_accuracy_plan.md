# 重新对比分析与精度提升计划：1_h264.mp4 目标 100%

## 1. 背景与目标
之前针对 H.264 解码器精度提升遇到了瓶颈。为突破当前 19.19% 精度且第 1 帧就发生不一致的问题，我们将**彻底忽略过去的结论和未验证的猜想**，从零开始，重新进行一场严密的交叉对比分析。
**对比基准**: `data/1_h264.mp4` (对比工具 FFmpeg)。
**参考源码**: `data/tmp_ffmpeg_src`, `data/tmp_openh264_src`, `data/tmp_vlc_src` (主要参考 FFmpeg 和 OpenH264，辅助参考 VLC)。
**最终目标**: 1_h264.mp4 逐帧对比精度（对比 FFmpeg）达到完全 100%。

## 2. 问题现状 (截止新计划启动时)
- **输入文件**: `data/1_h264.mp4`
- **当前精度**: `19.194623%`
- **报错首次出现**: `第 1 帧` (0为第一帧即I帧匹配通过，第 1 帧为首个P或B帧失败)

## 3. 分析与迭代方法论

### 第一阶段：定位异常点与变量对比分析 (对齐宏块解析)
我们必须精确找出是在哪个具体模块（语法解析？残差解码？运动矢量预测？插值？去块滤波？）引发了误差。
1. **获取 FFmpeg 日志**: 开启 FFmpeg 的 TRACE 级别调试信息，或者在 FFmpeg 源码中临时注入 printf。重点打印出 `第 1 帧` 的关键宏块状态（mb_type, MV, CBP, qp 等）。
2. **提取 Tao 引擎日志**: 使用 `TAO_H264_COMPARE_INPUT=data/1_h264.mp4 TAO_H264_COMPARE_FRAMES=2 TAO_H264_SLICE_TRACE=1 ...` 输出相应的宏块解析状态。
3. **文本/脚本对比分析**: 通过脚本自动化比对 Tao 与 FFmpeg 对同一宏块解析得到的值，找到出入点（如第一个解析错的 MV, 或 CBP 不一致的宏块）。
4. **源码查证**: 一旦定位到异构差异字段，深入 `data/tmp_ffmpeg_src/libavcodec/h264*` 和 `data/tmp_openh264_src/codec/decoder/core/src/` 中对应逻辑，查明差异。

### 第二阶段：像素级差异排查 (插值、预测与残差计算)
1. 如果语法解析（参数集/Slice Header/宏块预测模式和MV）表现均与开源实现一致，但解码依然失败，则可能出在像素计算上（如 CABAC解码、IDCT、插值处理或权重预测）。
2. 停用不必要的后处理：关闭去块滤波 (`TAO_SKIP_DEBLOCK=1 TAO_H264_COMPARE_SKIP_DEBLOCK=1`) 和其他后处理过程进行对比，确认误差是否由核心预测逻辑导致。
3. 精确对比每个宏块重构后的像素。

### 第三阶段：循环迭代修改与测试验证
- **制定假设**: 基于源码对比制定修改。
- **验证假设**:
  每次发现和修复问题后，运行 `TAO_H264_COMPARE_INPUT=data/1_h264.mp4 cargo test --test run_decoder h264::test_h264_compare_sample_1 -- --ignored` 检测。
- **5项严控**: 若检测到精度实打实提升（如突破 19% 到更高或者第一帧完全一致），必须在本地按照标准格式完成 `5项严控流程`，总结问题根本原因并 commit & push 代码，以稳定进度，切勿大幅改动不提交。

## 4. 第一步行动计划：诊断第 1 帧错误
由于 Frame 0 (I-Slice) 完全匹配，错误出现在 Frame 1 (P/B-Slice)。
这意味着，如果是 P 或 B 帧，可能的错误区：
- MV 预测计算 (Motion Vector Prediction) 有误。(如参考帧列表排序、Colocated MV 处理有误)
- DPB (参考帧缓冲区) 管理出错，导致选错了参考像素。
- B帧专属功能错误 (Direct 预测、Weighted Prediction (加权预测))。
- 亚像素插值滤波器。

**下一步操作**:
1. 提取 `1_h264.mp4` 用 Tao 解码时的调试日志。发现了一个**极为关键的突破口**：
   - Frame 0 (I帧) 成功解码了所有 8160 个宏块，消耗了 `438358/438360` bits，完全同步。
   - Frame 1 (P帧, slice_type=0, poc=8) 包含 `108000` 个 bits，但我们的解码器仅消耗了 `14084` bits 就因为 `end_of_slice_flag=1` 而提前终止了！导致只解码了 188 个宏块。
   - 随后的 Frame 2 (B帧) 和 Frame 3 (B帧) 也分别在 334 个和 401 个宏块处提前终止。
   这说明**根本原因不是像素插值误差**，而是 **CABAC 解析在 P/B 帧时脱轨（Desynchronization）**，导致提早读到了 `end_of_slice_flag=1`！由于帧的大部分未被解码，才会出现巨大的差异。

**行动步骤调整**:
1. 不再需要花精力排查像素插值和去块滤波过滤，问题是100%发生在语法解析(CABAC)阶段。
2. 我们需要聚焦于 Frame 1 (P帧) 的 CABAC 解析过程。异常提前截断发生在 MB index = 187/188 附近（或更早处刚脱轨）。
3. 增加宏块解析级别的详细日志 (如记录每个 MB 的 mb_skip_flag, mb_type, mvd, ref_idx, cbp 等)。
4. 对比 FFmpeg 在 Frame 1 (POC=8) 前 188 个宏块的解析结果，寻找第一个出现解析分支不同的宏块。
5. 一旦找到差异宏块，排查我们的 `cabac.rs` / `macroblock_inter.rs` 中的上下文模型（ctx_idx）计算或语法元素提取逻辑是否与 FFmpeg 存在出入。
